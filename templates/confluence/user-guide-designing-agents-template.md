# User Guide - Designing Agents

## Who This Guide Is For

Use this guide if you have an idea for a ROVO agent, automation, prompt, or workflow improvement and need help finding the right-sized solution.

## Before You Start

You should know:

- The workflow you want to improve.
- Who uses the workflow.
- Where the source information lives.
- Who owns the process or source data.
- What success would look like.

## Step 1: Start With Fit Triage

Go to `Agent Fit Intake`, preferably through the ROVO Design Agent.

The best outcome may be:

- Prompt
- Automation
- Existing-agent extension
- Confluence template
- App candidate
- New governed agent
- Not ready

Creating a new agent is only one valid outcome.

## Step 2: Check Existing Agents

Review `Agent Inventory`.

Look for:

- Existing agent with similar purpose.
- Existing broad agent that can be extended.
- Existing pattern in Pattern Library.
- Similar Project Brain.

## Step 3: Check Context Readiness

Use `Context Readiness Checklist`.

Do not build an agent on top of unclear, stale, duplicated, or ownerless documentation.

## Step 4: Create Or Update Design Artifacts

For durable agent work, create or update:

- Agent Design Record.
- Project Brain.
- Knowledge Source Plan.
- Subagent Settings.
- Rovo Studio Configuration.
- Evaluation.
- Measurement plan.
- Readiness status.
- Change Log.

## Step 5: Review Governance

Use `Agent Governance Review Template`.

The review should confirm:

- Owner and backup owner.
- Purpose and audience.
- Scoped knowledge sources.
- Tools or skills with risk notes.
- Documented permissions.
- Risk controls.
- Evaluation coverage.
- Measurement plan.
- Project Brain link and last-reviewed date.

## Step 6: Launch And Measure

After launch:

- Track adoption.
- Estimate time saved.
- Capture quality and rework signals.
- Review feedback.
- Update docs and Studio config as needed.

## Common Mistakes

- Starting with `New governed agent` before checking inventory.
- Giving broad knowledge access without justification.
- Skipping source ownership.
- Skipping evaluation.
- Treating measurement as optional.
- Letting Studio config drift from Confluence documentation.
